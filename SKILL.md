# 寓瞳开发约束技能

## 适用场景

当用户提出以下需求时，立即应用本技能：
- 生成新代码、补全模块、创建文件
- 修复 Bug、改逻辑、重构方法
- 调整前端组件、样式、主题
- 设计或修改接口路由、URL、API 端点
- 输出代码说明、开发文档或改动说明

## 全局硬性规则

1. 始终使用简体中文输出（解释、提示、注释、日志、接口文案均为中文）。
2. 文件内容必须兼容 UTF-8，避免乱码。
3. 新增代码文件按职责分类存放，保持目录结构清晰，不把所有文件堆在根目录。
4. 文件名简短且语义明确，避免重名文件或重名类。

## 接口与路由规则

1. URL 与端点命名保持简短直接。
2. 禁止在 URL 或接口全路径中出现以下词：`backend`、`frontend`、`html`、`php`。

## 注释、日志与调试规则

1. 注释与业务提示全部中文。
2. 禁止提交调试废话（如“先写死”“测试用”）。
3. 业务日志需包含中文业务语义，不允许仅输出无上下文英文片段。

## 代码修改核心原则

1. 采用最小改动策略：仅修改解决问题所必需的部分。
2. 非必要不重命名已有变量、函数、类型。
3. 修复问题时不得误删原有关键分支、校验和业务保护逻辑。
4. 若发现改动可能影响旧流程，优先补充兼容判断，而不是直接替换旧逻辑。

## 前端组件与样式规则

1. 按钮、表格、弹窗、Toast、输入框、搜索框等核心组件必须复用全局公共组件或全局样式。
2. 若缺少全局样式，先在全局样式层补齐，再在页面引用，禁止页面内零散私有硬编码样式。
3. 禁止硬编码颜色值（如 `#FFFFFF`、`rgb(...)`），必须使用 CSS 变量（`var(--token)`）。
4. 新增样式必须同时兼容浅色与暗黑主题。
5. 每个页面必须有独立且一一对应的静态资源文件：页面文件、样式文件、脚本文件同名配对（如 `login.html`、`login.css`、`login.js`）。
6. 页面文件名、对应 `css/js` 文件名尽量短，优先单词命名；避免冗长多词和语义不清命名。
7. 每个页面的 `html/css/js` 必须存放在同名目录下（如 `login/login.html`、`login/login.css`、`login/login.js`）。
8. 每个页面文件顶部必须包含“文件中文名 + 英文名”注释，HTML 使用 `<!-- ... -->`，CSS/JS 使用 `/* ... */`。

## 执行流程（每次编码前）

1. 先识别任务类型（后端逻辑 / 接口 / 前端样式 / 文档）。
2. 从本技能中加载对应约束。
3. 设计最小改动方案并检查是否影响旧逻辑。
4. 落地改动后自检：
   - 是否全中文输出与注释
   - 是否遵守 URL 禁词
   - 是否复用全局组件与样式
   - 是否满足页面 `html/css/js` 同名一一对应
   - 是否满足页面目录与文件名同名归档
   - 是否已添加文件顶部中英文名称注释
   - 是否避免硬编码颜色
   - 是否保持旧逻辑完整

## 冲突处理优先级

当本技能与其他通用建议冲突时，优先级如下：
1. 用户明确指令
2. 本技能硬性约束
3. 项目现有代码风格
4. 通用最佳实践
