# 后端结构说明

## 1. 当前架构定位

`backend/Aura.Api` 已经完成从“宏单体集成”到“模块化解耦”的演进。目前 `Program.cs` 已实现极简，业务逻辑与切面逻辑通过扩展机制（Extensions）与独立中间件（Middleware）实现。

- **极简入口**：`Program.cs` 只负责宿主启动、中间件流水线编排和核心配置引导。
- **扩展机制**：利用 C# 扩展方法将服务注册、路由映射、安全配置拆分到独立文件。
- **中间件独立化**：横切面逻辑（如安全头、前端路由逻辑、身份预校验）由独立 Middleware 承载。

## 2. 核心拆分机制 (Extensions)

为了防止 `Program.cs` 过快膨胀，我们采用了基于功能职责的拆分机制：

### 2.1 ServiceExtensions (`Extensions/ServiceExtensions.cs`)
- **DI 注册中心**：统一管理所有应用服务（Scoped）、单例工具（Singleton）及 HostedService 的生命周期。
- **配置绑定**：集中处理认证（JWT）、数据库（PostgreSQL）、缓存（Redis）及告警器的初始化路径。
- **认证策略**：封装了完整的 JWT Bearer 认证配置与授权策略声明。

### 2.2 EndpointExtensions (`Extensions/EndpointExtensions.cs`)
- **路由拓扑**：定义了整个系统的 API 树状结构，包括业务分组（Group）与属性标签。
- **最小化逻辑**：端点闭包内部仅执行参数绑定与服务调用，辅以必要的频率限制（RateLimit）校验。
- **协议适配**：集成了多种抓拍协议（ISAPI / C++ SDK / ONVIF）的归一化转换逻辑。

## 3. 中间件独立化成果 (Middleware)

系统核心逻辑与请求流水线通过自定义中间件实现解耦：

- **`SecurityHeadersMiddleware`**：提供自动化的安全标头注入（CSP/HSTS/Nosniff/DENY），增强基础防御能力。
- **`FrontendRoutingMiddleware`**：
  - 处理单页应用（SPA）的静态路由重写。
  - 实现未登录状态下的自动重定向（/login）。
  - 处理前端 HTML 文件的路径自动匹配。

## 4. 主要服务目录与职责

### 4.1 接入与业务核心 (Capture)
- `Capture/CaptureProcessingService.cs`：抓拍链路核心生产线。
- `Capture/CaptureOpsService.cs`：运维相关抓拍查询与模拟。
- `Capture/Adapters/`：多厂商协议适配器。

### 4.2 研判与算法 (Judge & Clustering)
- `JudgeService.cs`：寝室研判算法（归寝/群租/夜不归宿）。
- `Clustering/`：特征聚类任务与应用层调度。
- `VectorApplicationService.cs`：跨服务特征检索编排。

### 4.3 基础支撑 (Data, Cache & Ops)
- `Data/PgSqlStore.cs`：PostgreSQL 归一化操作层。
- `Cache/RedisCacheService.cs`：提供分布式锁、限流器与重试队列。
- `Ops/EventDispatchService.cs`：封装 SignalR 的实时广播分发。
- `Ops/AlertNotifier.cs`：异步告警通知核心。

## 5. 现状与维护建议

### 测试现状
- 核心算法（聚类）与工具组件（导出）已有 `Aura.Api.Tests` 覆盖。
- 建议后续增加 `CaptureProcessingService` 的端到端模拟测试。

### 维护指导
- **新增模块**：应在 `Application/` 目录新建子文件夹存放 Service，并在 `ServiceExtensions` 中注册，在 `EndpointExtensions` 中映射。
- **改动风险**：修改 `FrontendRoutingMiddleware` 时需注意不要拦截 `/api` 路径，以免造成接口失效。
- **配置变更**：所有敏感配置（Key/Secret）应优先从环境变量读取，`appsettings.json` 仅保留结构。
