# 部署文档与运维手册

## 0. 文档适配版本

- 当前适配版本：`0.1.8`
- 若版本升级，请同步核对 `CHANGELOG.md` 后再执行部署。

## 1. 部署拓扑

- `backend/Aura.Api`：中枢 API（默认 `https://localhost:5001`）
- `ai/main.py`：AI 服务（默认 `http://127.0.0.1:8000`）
- `PostgreSQL`：业务主库
- `Redis`：缓存与重试队列
- `ArangoDB`：向量检索

## 2. 首次部署步骤

1. 导入数据库：`database/schema.pgsql.sql`
2. 启动 AI 服务：
   - `cd ai`
   - `python -m venv .venv`
   - `.\.venv\Scripts\Activate.ps1`
   - 若提示执行策略限制：`Set-ExecutionPolicy -Scope Process Bypass`
   - `python -m pip install --upgrade pip`
   - `pip install -r requirements.txt`
   - `python -m uvicorn main:app --host 127.0.0.1 --port 8000`
   - 退出虚拟环境：`deactivate`
3. 启动后端服务：
   - `cd backend/Aura.Api`
   - `dotnet run`
4. 直接访问：`https://localhost:5001/`（前端已挂载到后端静态目录）

## 3. 核心配置项

- `backend/Aura.Api/appsettings.json`
  - `ConnectionStrings:PgSql`
  - `ConnectionStrings:Redis`
  - `Ai:BaseUrl`
  - `Jwt:*`
  - `Security:HmacSecret`
  - `Security:CaptureIpWhitelist`
  - `Storage:CaptureRetryRoot`：AI 失败时抓拍重试落盘目录；留空则使用项目下 `storage/uploads/capture-retry`。多实例水平扩容时，**所有 API 实例必须挂载同一共享路径**（如 NFS），否则 `POST /api/retry/process` 可能读不到他机写入的临时文件。
  - `CaptureRetry:PreferInlineBase64`：设为 `true` 时**不落盘**，失败抓拍图以 Base64 写入 Redis 重试队列，适用于无共享盘的多实例（注意队列与内存体积，生产请结合监控）。

### 3.1 水平扩容与限流说明

- 抓拍网关限流在 Redis 可用时按 **设备 ID** 计数，避免内网 NAT 下同 IP 多设备互相挤占配额。
- 已登录接口（研判、导出、向量、重试处理等）在 Redis 可用时按 **JWT 主体（用户名）** 计数。
- 设备列表接口使用 Redis 短缓存；**设备注册成功**后会自动删除 `device:list` 缓存键，无需手工清理。

### 3.2 可观测性（Prometheus / OpenTelemetry，可选）

**Prometheus（指标）**

- 配置项 **`Ops:Metrics:ExposePrometheus`**（环境变量 **`Ops__Metrics__ExposePrometheus`**）：为 **`true`** 时暴露 **`GET /metrics`** 供抓取；集成测试环境 **`Testing`** 一般为 **`false`**。
- 开发环境默认仅 **`https://localhost:5001`**。本机用 Docker 跑 Prometheus 时，可在抓取配置中对宿主机使用 **`host.docker.internal:5001`**，并启用 **`tls_config.insecure_skip_verify: true`**（仅用于本机联调）。
- 最小 `prometheus.yml` 示例与 **`docker run prom/prometheus`** 命令见根目录 **`README.md`** 章节「本机可观测性最小示例（可选）」。
- 生产环境应对 **`/metrics`** 限制来源网段或仅在集群内抓取；Ingress 按路径拒绝公网访问的示例见 **`deploy/k8s/ingress-nginx-deny-public-metrics.example.yaml`**。

**OpenTelemetry（链路追踪，OTLP）**

- 配置段 **`Ops:Telemetry`**：将 **`EnableTracing`** 设为 **`true`** 且配置 **`OtlpEndpoint`**（或环境变量 **`OTEL_EXPORTER_OTLP_ENDPOINT`**）后，向 OTLP 采集端导出 ASP.NET 与 **`HttpClient`** 追踪；默认关闭，避免无采集端时空跑。
- **`OtlpProtocol`**：**`Grpc`**（默认，常见端口 **4317**）或 **`HttpProtobuf`**。
- **`ServiceName`**：可选，默认 **`Aura.Api`**；也可用 **`OTEL_SERVICE_NAME`**。
- 本机最小示例（Jaeger **all-in-one**）与 **`.env`** 中 **`Ops__Telemetry__*`** 写法见根目录 **`README.md`** 同章节。

**维护约定**：若在仓库根目录 **`.env`** 中增加上述键，请同步更新 **`.env.example`**（仅占位符，勿提交真实内网采集端地址若涉密）。

## 4. 日常运维

- 健康检查：`GET /`
- AI 健康巡检重点字段（`GET /`）：
  - `熔断状态`：查看检索熔断是否打开（`is_open`）及恢复时间（`open_until`）
  - `限流状态`：查看每分钟限额（`limit_per_minute`）、当前分钟已用（`current_requests`）、剩余额度（`remaining`）
  - `回填状态`：查看历史桶字段回填轮次（`rounds`）、条数（`rows`）、失败次数（`failures`）、总体状态（`status`）
- 检索审计日志：`GET /ai/search-audit-logs?limit=100`（结构化 JSON，默认用于追查检索慢请求、空结果与失败原因）
- 重试队列状态：`GET /api/retry/status`
- 重试任务处理：`POST /api/retry/process`
- 操作日志巡检：`GET /api/operation/list`
- 导出文件目录：`storage/outputs`
- 上传文件目录：`storage/uploads`

### 4.1 巡检清单与示例响应

1. 先调用 `GET /`，确认 `code=0` 且 `model_loaded=true`。
2. 重点确认 `熔断状态.is_open=false`，若为 `true`，需先排查下游向量检索可用性。
3. 重点确认 `限流状态.remaining` 不长期为 `0`，避免高峰期持续限流。
4. 重点确认 `回填状态.status` 为「正常」，若为「异常」需检查 ArangoDB 与回填任务日志。
5. 调用 `GET /ai/search-audit-logs?limit=100`，按 `status` 与 `latency_ms` 排序排查异常请求。
6. 建议直接执行脚本自动判定：`powershell -ExecutionPolicy Bypass -File "e:\Aura\AI检索巡检脚本.ps1"`。

## 5. 联调与压测

- 执行脚本：`全系统联调与压测脚本.ps1`
- 脚本覆盖：登录、批量抓拍、研判触发、外联输出、导出校验
- AI 专项巡检：`AI检索巡检脚本.ps1`（默认检查 `GET /` 与 `GET /ai/search-audit-logs`，退出码：`0=通过`、`2=未通过`、`3=执行异常`）
- AI 专项巡检（CI JSON 模式）：`powershell -ExecutionPolicy Bypass -File "e:\Aura\AI检索巡检脚本.ps1" -JsonOutput`（仅输出 JSON，不输出彩色文本，适合 CI 采集）

## 6. 异常处理建议

- PostgreSQL 异常：系统自动回退内存模式，优先恢复数据库后再执行数据回灌
- Redis 异常：重试队列失效但主流程可运行，恢复后执行补偿
- AI 异常：抓拍仍入库，失败任务进入重试队列
- ArangoDB 异常：向量检索降级为内存索引
- Uvicorn 在个别环境下 `--reload` 可能不稳定，建议生产与联调默认使用不带 `--reload` 的启动方式

## 7. 安全建议

- 生产环境必须替换默认 JWT Key 与测试账号密码
- 仅开放必要端口，限制来源网段
- 保留操作日志与导出日志，满足审计需求
